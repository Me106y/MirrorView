# CareerForge — AI 求职工具套件架构文档

## 1. 产品概述

CareerForge 是一套面向求职者的 AI Skill 工具组合，覆盖从简历分析到面试准备的完整求职流程。包含四个独立但可协同工作的 Skill：

| Skill | 名称 | 功能定位 |
|-------|------|--------|
| resume-match | 简历匹配度分析 | 上传简历 + JD → 多维度评分 + 匹配等级 + 优化建议 |
| resume-craft | 简历生成与优化 | 从零生成或优化已有简历 → HTML + PDF 双格式输出 |
| cover-letter | 求职信撰写 | 基于简历 + JD 生成针对性 Cover Letter |
| mock-interview | 模拟面试 | 三轮模拟面试 + 面试报告 + 逐题反馈 |

**核心设计原则：**
- 每个 Skill 独立可用，也可串联使用（如 resume-match 发现问题 → resume-craft 优化 → cover-letter 撰写 → mock-interview 准备）
- 所有文件输出统一为 HTML + PDF 双格式
- 支持中英文双语
- 简历严格控制在两页以内

## 2. Skill 1: resume-match（简历匹配度分析）

### 2.1 触发条件

用户提到以下关键词时触发：简历匹配、简历分析、岗位匹配、JD分析、简历评分、resume match、resume analysis、job fit、简历诊断、简历评估。用户同时提供了简历文件和岗位 JD（文字或图片）时自动触发。

### 2.2 输入要求

- **必须**：用户简历（PDF / DOCX / 文字粘贴）
- **必须**：目标岗位 JD（文字粘贴 / 图片上传 / URL）

### 2.3 分析维度（共6个维度，每个维度满分100分）

| 维度 | 权重 | 评估内容 |
|------|------|----------|
| 硬性技能匹配度 | 25% | JD 要求的具体技术/工具/技能 vs 简历中体现的技能 |
| 工作经验相关度 | 25% | 过往经验与目标岗位的职责、行业的关联程度 |
| 软性能力匹配度 | 15% | 沟通、管理、协作等软技能的匹配情况 |
| 教育背景匹配度 | 10% | 学历、专业、证书与岗位要求的契合度 |
| 关键词覆盖率 | 15% | 简历中覆盖了 JD 多少关键词/行业术语 |
| 简历质量 | 10% | 排版、结构、表达、数据量化、篇幅等 |

### 2.4 输出结构

```
┌─────────────────────────────────────────┐
│  整体匹配度评分：XX / 100               │
│  匹配等级：A / B / C                    │
├─────────────────────────────────────────┤
│  各维度评分详情（6个维度逐一展开）       │
│  ├─ 维度名称 + 分数 + 进度条            │
│  ├─ 具体分析（匹配了什么/缺少什么）     │
│  └─ 该维度的优化建议                    │
├─────────────────────────────────────────┤
│  关键差距分析                           │
│  ├─ JD 要求但简历缺失的关键项           │
│  └─ 简历中的优势但未被 JD 提及的项      │
├─────────────────────────────────────────┤
│  优化建议                               │
│  ├─ 简历措辞优化建议（逐条）            │
│  ├─ 能力提升建议（中长期）              │
│  └─ 优化后的完整简历（Markdown 格式）   │
└─────────────────────────────────────────┘
```

### 2.5 匹配等级判定标准

- **A. Strong Fit（≥75分）**：建议申请。简历与 JD 高度匹配，仅需微调措辞。输出：直接给出优化后简历。
- **B. Stretch Fit（50-74分）**：可以申请但需调整定位与措辞。输出：指出需要重新包装的经历，给出改写建议和优化后简历。
- **C. Poor Fit（<50分）**：不建议申请。输出：明确说明不匹配的核心原因（不是泛泛而谈），给出「如果仍想争取需要补足什么」的具体路径，以及建议用户考虑的其他方向。

### 2.6 技术实现要点

- 读取简历文件：使用 file-reading skill 读取 PDF/DOCX
- JD 解析：支持纯文字、图片 OCR、URL 抓取
- 输出格式：在对话中以结构化方式呈现分析结果，同时生成 Markdown 格式的优化简历
- 可选：生成可视化评分报告（HTML artifact）

## 3. Skill 2: resume-craft（简历生成与优化）

### 3.1 触发条件

用户提到以下关键词时触发：生成简历、写简历、做简历、简历制作、优化简历、改简历、简历美化、resume create、resume build、resume optimize、帮我做一份简历、简历排版、简历模板、resume template、简历设计、resume design。当用户从 resume-match 衔接过来（已有优化后的 Markdown 简历），也应触发此 Skill 来生成 HTML+PDF 版本。

### 3.2 工作流程

触发后，AI 首先判断用户属于哪条路径：

```
用户触发 resume-craft
    │
    ├─ 路径 A：从零生成
    │   └─ 进入结构化提问流程（含模板选择）
    │
    └─ 路径 B：优化已有简历
        └─ 读取简历 → 确认优化方向 → 选择模板 → 生成
```

### 3.3 路径 A：从零生成 — 结构化提问清单

AI 依次收集以下信息（分批提问，不要一次问完）：

**第一轮：基础信息 + 模板选择**
- 目标岗位是什么？（如有 JD 请提供）
- 生成中文简历还是英文简历？是否需要同时生成中英文两版？
- 是否需要在简历上放照片？（如需要请上传）
- 选择简历模板风格（展示 7 种选项）：

| 编号 | 模板名称 | 风格特点 |
|------|---------|---------|
| 01 | **Editorial 杂志编辑风** | 经典文艺，奶油色底，绿色强调线，适合创意/文化类岗位 |
| 02 | **Minimal 极简主义** | 纯白底，极少装饰，留白多，适合科技/设计/外企 |
| 03 | **Sidebar Navy 深蓝双栏** | 左侧深蓝栏放技能和联系方式，右侧放经历，信息密度高 |
| 04 | **Sidebar Dark 深灰左栏** | 左侧深灰栏+右侧白底，沉稳大气，适合管理/金融类 |
| 05 | **Dark Header 深色头部** | 顶部深色块放姓名和联系方式，正文白底，对比醒目 |
| 06 | **Clean Teal 清新青色** | 白底+青绿色条装饰，清新专业，适合大部分岗位 |
| 07 | **Elegant 优雅对称** | 居中对称排版，衬线体为主，适合学术/高管/传统行业 |

如果用户有自己喜欢的配色或强调色，也可以自定义——只需告知偏好颜色，在选定模板基础上替换即可。

**第二轮：个人信息**
- 姓名、联系方式（手机、邮箱）、所在城市
- 是否需要显示其他信息（LinkedIn、GitHub、个人网站等）

**第三轮：教育背景**
- 学校、专业、学位、在读时间
- GPA、奖学金、荣誉奖项
- 是否有多段教育经历

**第四轮：工作/项目经历（核心环节 — AI 深挖模式）**
- 每段经历：公司/组织、职位、时间
- 核心职责和成果（引导用户用数据量化）
- 是否有与目标岗位相关的项目经历

⚠️ **AI 深挖（Grill）机制：** 收集到用户的经历信息后，AI 需要对照目标 JD 进行匹配度分析。如果发现：
  - **信息量不足**：用户的描述过于笼统，缺乏具体数据和成果 → AI 继续追问细节
    - 「这个项目的具体成果是什么？有没有可量化的数据？」
    - 「你在团队中具体负责哪个环节？」
    - 「遇到过什么挑战？你是怎么解决的？」
  - **与 JD 匹配度不够**：现有经历和目标岗位关联性弱 → AI 引导用户换角度挖掘
    - 「你之前的工作中有没有涉及XX方面的经验？哪怕是很小的部分？」
    - 「你有没有做过任何和XX相关的个人项目、学习或探索？」
    - 「你日常工作中用过XX工具/方法吗？哪怕不是主要职责？」
  - **可以更好表述**：经历本身有价值但表述未突出亮点 → AI 主动建议重新表述（忠于事实，只改表达方式）
    - 「你说的这段经历其实可以用XX角度来描述，和目标岗位更匹配，你觉得准确吗？」

深挖过程自然融入对话，不要让用户感觉在被审问。每段经历最多追问 2-3 次，用户说"没有了"就不再纠缠。

**第五轮：技能与证书**
- 专业技能、工具
- 语言能力
- 证书、资质

**第六轮：确认与偏好**
- 是否有需要特别突出的经历或技能
- 确认所有信息无误 → 开始生成

### 3.4 路径 B：优化已有简历

1. 读取用户上传的简历文件（PDF / DOCX / 文字 / Markdown）
2. 询问优化方向（可多选）：换岗位方向 / 美化排版 / 针对特定 JD 优化 / 内容精简
3. 选择简历模板风格（展示 7 种选项，同路径 A）。如果用户对原有排版满意只想优化内容，可以跳过
4. 询问：是否需要在中文简历优化定稿后，同时生成对应的英文版简历？
5. 如有目标 JD，自动进行匹配度分析（参考 resume-match 的逻辑）
6. 列出具体优化方案，等用户确认后再生成
7. 生成中文简历
8. 如用户需要英文版：中文版定稿确认后，基于定稿内容生成英文版

### 3.5 输出规范

**双格式输出：**
- HTML 文件：可在浏览器中查看，保留设计感，内置导出 PDF 按钮
- PDF 文件：通过 Playwright 从 HTML 生成（后台），或用户通过 HTML 中的导出按钮自行生成（`window.print()`）

**7 种模板风格：**

每种模板有独立的字体搭配、配色方案和布局特征，详见 `references/design-system.md`。用户也可在选定模板基础上自定义强调色。

**排版硬性规则：**
- 严格控制在两页以内（A4 纸）
- 如果内容超出两页：
  1. 先告知用户，列出哪些部分可以精简，询问用户是否同意
  2. 用户同意精简 → 按用户意见删减内容
  3. 用户不同意精简 → 适当缩小字号和间距来适配两页，但正文字号不得低于 11px
- 间距标准：section 间距 20-24px，bullet 间距 1-4px，行高 1.55-1.7
- 照片：单栏模板放右上角（3:4 长方形），双栏模板放左侧栏顶部（圆形）

**内容结构顺序（按目标岗位调整优先级）：**
1. 姓名 + 职位标题 + 联系方式（+ 可选照片）
2. 个人简介（3-5行，突出核心优势）
3. 与目标岗位最相关的经历（项目经验 / 工作经历，根据情况调整顺序）
4. 其他工作经历
5. 教育背景
6. 技能与工具

**双语简历规则：**
- 先定稿中文版，用户确认后再生成英文版
- 英文版不是逐字翻译，而是按英文简历的表达习惯重新撰写（action verbs 开头、避免中式英语）
- 两版简历保持相同的视觉设计，仅替换文字内容和字体

### 3.6 HTML 导出功能（所有模板必须包含）

每个生成的 HTML 简历都内置导出 PDF 按钮：

- **导出方式**：`window.print()`（浏览器原生打印），生成可选中、可搜索、可被 HR 系统解析的真文字 PDF
- **CSS 打印规则**：`@page { size: A4; margin: 0; }` + `print-color-adjust: exact`
- **分页控制**：
  - `.job` 和 `.section` 允许跨页（避免大段空白）
  - `.project-card`、`.edu-item`、`.job-header` 禁止截断
  - `.section-title` 和 `.job-header` 加 `page-break-after: avoid`（防止标题孤行）
- **两页压缩**：在 `@media print` 中添加字号/间距缩小规则，确保 PDF 严格两页以内
- 按钮旁提示用户：另存为 PDF → 纸张 A4 → 边距「无」→ 勾选「背景图形」

### 3.7 技术实现要点

- **模板参考**：`templates/resume-template.html`（Editorial 风格完整实现）
- **视觉预览**：`templates/CareerForge-模板预览.html`（7 种模板的视觉效果）
- **设计规范**：`references/design-system.md`（7 种模板的详细字体/配色/布局规范）
- **PDF 生成**（后台）：`scripts/generate_pdf.py`，使用 Playwright 将 HTML 转为真文字 PDF
- **PDF 导出**（前端）：HTML 内置 `window.print()` 按钮，用户可自行导出
- **照片处理**：`scripts/process_photo.py`，使用 PIL 裁剪为 3:4 比例，压缩为 base64 嵌入 HTML
- 如果 Playwright 未安装，提示用户运行：`pip install playwright && playwright install chromium`
- 如果 PIL 未安装，提示用户运行：`pip install Pillow`

## 4. Skill 3: cover-letter（求职信撰写）

### 4.1 触发条件

用户提到以下关键词时触发：cover letter、求职信、自荐信、申请信、写一封求职信、application letter、帮我写个求职信、投递邮件怎么写、怎么跟 HR 打招呼、boss直聘打招呼、招呼语、开场白、自我介绍信。当用户从 resume-match 或 resume-craft 衔接过来（已有简历和 JD 信息），也应触发。

### 4.2 输入要求

- **必须**：目标岗位 JD（文字 / 图片 / URL）
- **推荐**：用户简历（可从 resume-craft 衔接，也可直接上传）
- **可选**：公司名称、具体申请渠道（邮件 / 系统 / 内推）、目标语言（中文 / 英文）

如用户未提供简历，AI 通过提问收集关键信息：最相关的 2-3 段经历、核心技能、申请动机。

### 4.3 生成逻辑

**前置提问：** AI 首先询问用户的申请渠道：
- **场景 A：邮件投递** → 生成完整信件格式（含抬头、日期、称呼、落款）
- **场景 B：招聘软件打招呼** → 生成简短的打招呼消息（开门见山，3-5句话）

**核心原则：Cover Letter ≠ 简历复述。Cover Letter 的作用是讲故事、建立连接。**

AI 从简历中提取与 JD 最匹配的 2-3 个亮点，根据场景生成：

**场景 A：邮件投递（完整信件格式，300-500字）**
```
第一段：开场（为什么对这个岗位/公司感兴趣 — 需要有真实感）
第二段：核心匹配（从简历中提取最匹配 JD 的 1-2 个经历展开，强调成果）
第三段：独特价值（简历里不容易体现的软性优势，如跨领域背景、学习能力）
第四段：收尾（表达期待，简洁有力）
```

**场景 B：招聘软件打招呼（精简消息，80-150字）**
```
一句话说明身份和意图
一句话点出最匹配的核心优势
一句话表达期待进一步沟通
```

### 4.4 输出格式

**纯文本输出，不生成 HTML/PDF。** 直接在对话中呈现，用户可一键复制。

| 场景 | 格式 | 字数 |
|------|------|------|
| 邮件投递 | 完整信件（含称呼、落款、日期） | 300-500字 |
| 招聘软件打招呼 | 短消息 | 80-150字 |

### 4.5 质量标准

- 不使用空洞的套话（如「我是一个积极向上的人」）
- 每个论点都有简历中的具体经历支撑
- 体现对公司/岗位的理解（如果 AI 能搜索到公司信息，主动融入）
- 语气专业但不僵硬，有个人特色
- 中文 Cover Letter 和英文 Cover Letter 不是互相翻译，而是分别按各自的文化习惯撰写

## 5. Skill 4: mock-interview（模拟面试）

### 5.1 触发条件

用户提到以下关键词时触发：模拟面试、面试准备、面试练习、面试辅导、mock interview、interview prep、面试模拟、帮我准备面试。

### 5.2 输入要求

- **必须**：目标岗位 JD（文字 / 图片 / URL）
- **必须**：用户简历（PDF / DOCX / 文字粘贴，或从 resume-craft 衔接）
- **可选**：公司名称（用于搜索公司背景信息和面试经验）、面试轮次（用户可选择只做某一轮）

### 5.3 面试流程设计

```
准备阶段
  │  ① AI 分析 JD + 简历，识别用户的强项和弱项
  │  ② AI 搜索公司背景信息（如提供了公司名）
  │  ③ AI 在网上搜索该公司/该岗位的面试经验帖
  │     （如知乎、脉脉、牛客、Glassdoor 等平台的面经）
  │     如果搜到了，参考真实面经中的高频问题融入题库
  │  ④ 基于以上信息生成三轮面试题库
  │  ⑤ 向用户说明面试流程和规则
  │
第一轮：HR 面试（5-6题）
  │  考察维度：求职动机、文化匹配、稳定性、薪资期望、职业规划
  │  风格：表面友好，但会在关键问题上持续追问
  │  追问逻辑：
  │    - 用户说「想换个环境」→ 追问「具体是什么让你不满意？」
  │    - 用户说「对这个方向感兴趣」→ 追问「做过什么来证明这个兴趣？」
  │    - 用户给出模糊回答 → 要求给具体例子
  │  压力面场景：
  │    - 「你的简历显示你之前做的是XX，为什么觉得自己能做XX？」
  │    - 「如果入职后发现和你想象的不一样，你会怎么办？」
  │    - 「你期望薪资是多少？如果我们给不到呢？」
  │
第二轮：业务主管面试（6-8题）
  │  考察维度：专业能力、项目经验、技术深度、问题解决
  │  风格：直接、细节导向、层层深挖
  │  深挖逻辑（针对项目经历，至少追问3层）：
  │    第1层：「讲讲你做的XX项目」
  │    第2层：「遇到最大的困难是什么？你怎么解决的？」
  │    第3层：「如果重新来过，你会怎么做不同？为什么？」
  │  压力面场景：
  │    - 针对简历弱项直接发问：「你没有XX经验，为什么觉得能胜任？」
  │    - 设置情景题考察应变：「如果项目上线后出了严重bug，你的处理步骤是什么？」
  │    - 故意质疑用户的方案：「你这个方案有XX问题，你怎么看？」
  │    - 技术细节追问：「你说用了XX技术，能解释一下它的原理/你为什么选它？」
  │
第三轮：终面 / 高管面试（4-5题）
  │  考察维度：战略思维、学习能力、团队协作、成长潜力、大局观
  │  风格：开放性问题为主，看思维方式而非标准答案
  │  深挖逻辑：
  │    - 用户说了一个观点 → 提出反面论点要求回应
  │    - 用户描述了成功经验 → 问失败经验
  │  压力面场景：
  │    - 「坦白说，你的背景和我们大多数候选人差距很大，你觉得你的优势在哪？」
  │    - 「如果你的方案被团队否决了，你会怎么做？」
  │    - 沉默几秒后说「还有吗？」（测试用户是否能补充更深层的思考）
  │
面试结束
  └─ 生成完整面试报告
```

### 5.4 面试真实感规则

**追问与深挖原则：**
- 每轮面试中至少有 2 个问题要进行追问（不是所有问题都追问，模拟真实节奏）
- 追问的触发条件：用户回答过于笼统/含糊、用户提到了一个有趣的点值得展开、用户的回答存在逻辑漏洞
- 追问最多 3 层，避免变成审讯

**压力面原则：**
- 每轮面试中设置 1-2 个压力面问题
- 压力面不是为了刁难，而是模拟真实面试中面试官会质疑的点
- 重点针对：简历中的短板、转行的合理性、经验不足的领域
- 语气直接但不冒犯，如「这个回答没有回答到我的问题，能再具体一些吗？」

**节奏控制：**
- 不是每个问题都高压，要有松有紧
- 开场 1-2 题用来暖场，中间穿插压力题，最后给用户一个「你有什么想问我的吗」的缓冲

**用户回答后不立即给反馈**（模拟真实面试体验）
- AI 可以根据用户回答进行追问
- 用户可以随时说「跳过」进入下一题
- 用户可以随时说「结束面试」提前生成报告
- 每轮结束时简短过渡（如「感谢你的回答，接下来由我们的业务负责人来和你聊聊」）

### 7.3 PDF 生成方案

**两种方式并存：**

1. **后台生成**（Playwright）：Claude Code 在后台调用 `scripts/generate_pdf.py`，生成真文字 PDF，用户无需操作
2. **前端导出**（window.print）：HTML 文件内置导出按钮，用户点击后通过浏览器打印功能自行生成 PDF

两种方式底层都是 Chrome 打印引擎，效果一致，生成的 PDF 文字可选中、可搜索、可被 HR 系统（ATS）解析。

## 8. 用户体验设计原则

1. **不要一次问太多问题**：每次最多问 2-3 个问题，分批收集信息
2. **给出示例引导回答**：特别是让用户描述工作经历时，给出「好的描述」vs「一般的描述」的对比
3. **保持对话感**：不要像填表，要像和一个有经验的职业顾问聊天
4. **主动提供建议**：不要只是被动收集信息，发现可以优化的地方要主动指出
5. **即时反馈**：生成结果后主动问「哪里需要调整」，支持多轮迭代
6. **尊重用户选择**：不要过度推销功能（如用户只想做面试练习，不要强推简历优化）

## 10. 注意事项

- 简历中的个人信息（手机号、邮箱）属于隐私数据，不应被记录或缓存
- 面试模拟中 AI 应保持专业中立，不应给用户施加不必要的心理压力
- 所有评分和建议应基于客观分析，不应带有歧视性（年龄、性别、学历歧视等）
- 当用户的期望与实际匹配度差距较大时（C级），应以鼓励但诚实的语气沟通，给出建设性的成长路径而非简单否定
